# AI System Design — Concepts Index

Last updated: 2026-07-04

Each entry: what it is, where it shines, where it fails, real-world notes, and at least one citation.

## Retrieval-Augmented Generation (RAG)

**What it is:** A pattern that augments an LLM's response by first retrieving relevant context from an external knowledge store (typically a vector database) and including that context in the prompt. The model generates an answer grounded in the retrieved context rather than relying on parametric memory alone.

**Where it shines:** Knowledge-intensive tasks where the answer depends on specific facts not reliably in the model's training data — internal documentation, recent news, proprietary product information, customer-specific data. RAG dramatically reduces hallucination rates on factual questions because the model can "look up" the answer rather than reconstruct it from memory.

**Where it fails:** When the retrieval step fails to return the right documents (the answer exists in the corpus but retrieval doesn't find it), the LLM generates an answer from missing context. This fails silently — the model produces a plausible-sounding answer with no indication that the retrieval was wrong. Also fails when the answer genuinely doesn't exist in the corpus and the model extrapolates rather than saying "I don't know."

**Real-world notes:** The quality of a RAG system is roughly: 70% retrieval quality, 30% generation quality. Investing in chunking strategy, hybrid retrieval, and a reranker is more impactful than switching from GPT-4o-mini to GPT-4o. First introduced systematically by Lewis et al. (NeurIPS 2020); now the dominant architecture for knowledge-intensive LLM applications.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Lewis et al., "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks," NeurIPS 2020. https://arxiv.org/abs/2005.11401

## Embeddings (Text)

**What it is:** A dense vector representation of text that captures semantic meaning. Similar texts produce vectors with high cosine similarity. Generated by an encoder model (e.g., OpenAI text-embedding-3-small, Sentence-BERT). Dimensionality typically ranges from 384 (small, fast) to 3,072 (large, accurate). Each dimension is a float32 (4 bytes).

**Where it shines:** Semantic similarity search — finding documents that are conceptually related to a query even when they share no keywords. Powers retrieval in RAG, semantic deduplication, clustering, and recommendation.

**Where it fails:** Exact-match queries where the query term is rare in the embedding model's training distribution (e.g., internal acronyms, product SKUs, person names). Also fails when the embedding model version changes — vectors encoded with model version A and vectors encoded with model version B are not comparable, causing retrieval to silently degrade.

**Real-world notes:** text-embedding-3-small (1,536 dimensions) is the default choice for cost/performance at startup scale. text-embedding-3-large (3,072 dimensions) provides ~5% better accuracy at 2× cost. For domain-specific corpora (medical, legal, code), fine-tuning an embedding model on domain-specific pairs can provide 10–20% retrieval improvement. Storage: 1,536 dims × 4 bytes = 6,144 bytes per vector.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Nils Reimers and Iryna Gurevych, "Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks," EMNLP 2019. https://arxiv.org/abs/1908.10084

## Approximate Nearest-Neighbor (ANN) Search / HNSW

**What it is:** An algorithm for efficiently finding the k most similar vectors to a query vector in a large vector space, without exhaustively comparing every vector. The dominant algorithm is HNSW (Hierarchical Navigable Small World graphs), which builds a multi-layer graph structure enabling O(log N) search instead of O(N).

**Where it shines:** Large-scale similarity search (millions to billions of vectors) where exact search is too slow. HNSW offers excellent recall (typically >95% of what exact search would return) at 10–1000× speedup.

**Where it fails:** Requires significant memory to store the graph structure (Pinecone's overhead is ~40 bytes per vector for the HNSW structure, on top of the vector itself). Also: HNSW's recall can degrade if the graph is not built with appropriate `ef_construction` and `m` parameters for the expected query pattern. Self-hosted pgvector's HNSW can degrade at >500K vectors without expert tuning.

**Real-world notes:** Pinecone, Weaviate, Qdrant, and Chroma all use HNSW internally. The "approximate" is important: you will not always get the exact top-k most similar vectors. For a startup-scale system (15K–150K vectors), this approximation error is negligible. At billion-scale, it requires careful calibration. FAISS (Meta's library) provides both exact and approximate search modes and is the reference implementation for understanding the algorithms.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Malkov and Yashunin, "Efficient and Robust Approximate Nearest Neighbor Search Using Hierarchical Navigable Small World Graphs," IEEE TPAMI 2020. https://arxiv.org/abs/1603.09320

## Hybrid Retrieval (Dense + Sparse / BM25)

**What it is:** A retrieval strategy that combines dense vector similarity (embedding-based) with sparse keyword matching (BM25 or TF-IDF) into a single ranked result. The combination is typically a weighted sum: `final_score = α × dense_score + (1-α) × sparse_score`, where α is tuned between 0 (pure sparse) and 1 (pure dense).

**Where it shines:** Corpora with mixed query types — some queries are semantic ("what is our philosophy on technical debt?") while others are exact-match ("what does RBAC stand for?"). Hybrid retrieval handles both; pure dense or pure sparse handles only one type well.

**Where it fails:** Requires fitting a BM25 sparse encoder on the full corpus before search — this adds complexity to the ingestion pipeline and must be kept in sync with the vector index. The alpha hyperparameter requires tuning; wrong values can degrade one query type while improving the other.

**Real-world notes:** Pinecone's native hybrid search uses the BM25Encoder from the `pinecone-text` library. Weaviate also supports hybrid search natively. In practice, a 0.75 alpha (75% dense) works well for general-purpose knowledge bases; lower alpha (more sparse weight) helps for highly technical or jargon-heavy corpora. The improvement over pure dense is typically 5–15% on recall@k metrics for corpora with acronyms, product names, or technical terminology.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Pinecone, "Hybrid Search Documentation." https://docs.pinecone.io/guides/data/query-sparse-dense-vectors

## Chunking

**What it is:** The process of splitting source documents into retrieval-sized units (chunks) that are individually embedded and stored in the vector index. The chunk is the unit of retrieval — not the document. Chunking parameters include: chunk size (target token count), chunk overlap (tokens shared between adjacent chunks), and splitting strategy (character, sentence, paragraph, or semantic boundary).

**Where it shines:** Necessary for any corpus where documents are longer than what can fit in a single embedding context window (typically 8,192 tokens for text-embedding-3-small). Without chunking, you'd need to embed entire documents, which averages over all their content and loses precision for specific passages.

**Where it fails:** Naive fixed-size chunking splits documents at arbitrary boundaries, potentially cutting mid-sentence or mid-list. The optimal chunk size is empirical and corpus-dependent — a chunk size that works for narrative prose may be too large for dense technical reference material and too small for conversational threads.

**Real-world notes:** 512 tokens with 64-token overlap is a good starting point for general-purpose documentation. Chunk size should be validated against a retrieval accuracy benchmark (gold-standard query set) at multiple sizes. "Small-to-big" retrieval is an advanced pattern: embed small chunks (128 tokens) for precise retrieval, but return the larger parent chunk (512 tokens) to the LLM for better context — gets the best of both worlds. Every chunk must carry document metadata (title, url, author) as payload, or citations become impossible.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** LangChain documentation, "Text Splitting." https://python.langchain.com/docs/how_to/recursive_text_splitter/

## Reranking (Cross-Encoder)

**What it is:** A second-stage relevance scoring step that takes the top-k candidates from an ANN retrieval step and re-scores them using a cross-encoder model. Unlike a bi-encoder (which encodes query and document independently and computes cosine similarity), a cross-encoder sees the query and document concatenated as a single input and produces a relevance score. This is more accurate but cannot scale to the full corpus (too slow).

**Where it shines:** When ANN retrieval top-k includes candidates of mixed relevance and you need to identify the truly relevant subset. Particularly effective for queries where word-level matching (BM25) and semantic matching (dense) both produce plausible but suboptimal rankings.

**Where it fails:** Adds 50–200ms latency and cost per query. Not suitable as a primary retrieval mechanism over large corpora (would be O(N) over the full index). Only useful when the initial retrieval is already reasonably good — a reranker cannot rescue retrieval that returned the wrong documents entirely.

**Real-world notes:** Cohere Rerank is the most commonly used hosted reranker (~$0.001 per call). Cross-encoder models can also be self-hosted (smaller, faster models exist — ms-marco-MiniLM-L-6-v2 is popular). The standard pattern: retrieve top_k=20 with ANN, rerank to top_k=8 for the LLM prompt. Using top_k=8 directly from ANN (without reranking) vs. reranking top_k=20 down to top_k=8 typically shows 10–15% improvement in answer quality for ambiguous queries.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Nogueira et al., "Passage Re-ranking with BERT," 2019. https://arxiv.org/abs/1901.04085

## Embedding Drift

**What it is:** The failure mode where the embedding model version used at query time differs from the version used when the index vectors were created. Because different model versions produce vectors in different geometric spaces, cosine similarity scores between new query vectors and old document vectors become unreliable. Retrieval quality degrades silently with no infrastructure error.

**Where it shines (as a concept):** Understanding this failure mode is essential for any production RAG system. It affects every system that uses a managed embedding model API without version pinning.

**Where it fails (as a concern):** Embedding APIs from major providers (OpenAI, Cohere) are relatively stable — OpenAI has not made breaking embedding model changes frequently. But the risk is real and has occurred. The cost of a full re-embedding (when it happens) is typically low at startup scale (~$0.09 for 15K chunks), but the damage to user trust from degraded quality during the drift period is harder to recover from.

**Real-world notes:** Prevention is simple: pin the model version string. Detection requires an offline eval pipeline with a gold-standard query set and a retrieval accuracy metric — standard infrastructure monitoring (latency, error rates) cannot detect this. Remediation: blue-green re-embedding into a new Pinecone namespace, validate, then cut over.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

## Idempotency (in Ingestion Pipelines)

**What it is:** The property that running an operation multiple times produces the same result as running it once. In the context of a RAG ingestion pipeline, idempotency means: re-running the ingestion job on a document that hasn't changed produces no change to the vector index. This requires deterministic IDs (so the system knows which vectors to overwrite vs. create) and upsert semantics (overwrite existing vectors by ID rather than always appending).

**Where it shines:** Makes ingestion pipelines safe to retry after partial failure, which will happen in production. Also makes debugging straightforward — you can re-run any ingestion job in any order and converge to the correct index state.

**Where it fails:** Idempotency requires upfront design — specifically, deterministic chunk IDs. If you start with auto-generated IDs, migrating to hash-based IDs in production requires a full index rebuild. Idempotency also doesn't help with deletion tracking — you need a separate mechanism to detect and delete vectors for removed documents.

**Real-world notes:** The canonical pattern for chunk IDs: `SHA-256(canonical_document_url + ":" + str(chunk_index))`. "Canonical" URL is important — if a Confluence page is moved, its URL changes but its canonical ID (page_id in Confluence's API) doesn't. Upsert with deterministic IDs is one of the most important design decisions in a RAG system and one of the most commonly missed in prototypes.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

## Semantic Caching

**What it is:** A caching strategy where the cache key is based on the semantic similarity of the query (via its embedding), rather than the exact query string. Queries that are semantically equivalent ("what is our vacation policy?" and "how many vacation days do I get?") map to the same cache bucket and return the same cached answer, even though they are textually different.

**Where it shines:** Reduces latency dramatically for repeat question patterns (common in knowledge bases, where many employees ask variations of the same questions). Also reduces LLM API spend linearly with cache hit rate.

**Where it fails:** Cache invalidation is complex: when a document updates, all cached answers grounded in that document should be invalidated. This is difficult to track precisely at the embedding-key level. The practical compromise: use TTL-based expiration (e.g., 4 hours matching the ingestion cycle) and accept that cached answers may be up to one ingestion cycle stale.

**Real-world notes:** The cache key implementation: quantize each embedding dimension to 2 decimal places → serialize the vector to a string → SHA-256 hash. The quantization step creates a tolerance band that clusters semantically similar queries without requiring an exact embedding match. TTL should match the ingestion cycle so stale cache entries don't survive longer than the sync interval. GPTCache and Langchain's `CacheBackedEmbeddings` are popular implementations.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

## Guardrails

**What it is:** A set of checks applied at various points in the AI pipeline to ensure the system behaves safely, correctly, and within intended scope. In a RAG system, guardrails include: pre-retrieval moderation (blocking off-topic or harmful queries), access control filtering (ensuring users only see documents they're authorized to access), and post-generation groundedness checking (verifying the LLM's answer is supported by the retrieved context).

**Where it shines:** Prevents the system from producing authoritative-sounding wrong answers (groundedness), leaking restricted information (access control), or being misused for purposes outside its design scope (moderation). Especially important in systems where the answer is taken as authoritative (HR policies, legal compliance, medical intake).

**Where it fails:** Groundedness checking adds latency (~200ms for a mini model call) and cost. It catches explicit hallucinations but may miss subtle extrapolations that are technically grounded in retrieved text. Access control at the chunk level (rather than the document level) is complex — a chunk from a restricted document may be indistinguishable from a public one in the vector metadata if tagging is not precise.

**Real-world notes:** The OpenAI Moderation API is effectively free and catches most abuse cases in a pre-retrieval check. For groundedness, a lightweight LLM call (mini or Haiku) checking whether each claim in the generated answer is present in the retrieved context works well and is faster than re-running a full retrieval. Access control: Pinecone supports metadata filtering on queries, enabling per-user document visibility restrictions by tagging each chunk with an `access_level` field and filtering on query.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

## Prompt Injection

**What it is:** An attack where malicious instructions embedded in user-controlled content (in a RAG system: in a document that gets retrieved and inserted into the prompt) attempt to override the system's intended behavior. Because LLMs process retrieved document content and system instructions in the same context window, malicious content in a retrieved document can influence the model's behavior.

**Where it shines (as a threat):** Particularly relevant in systems where: the document corpus is editable by a broad user population (internal wikis, public-facing docs), the system has privileged actions (sending emails, executing code), or the system is customer-facing (where the attacker and the victim are different parties).

**Where it fails (as a defense):** Complete prevention is impossible with current LLMs — sufficiently sophisticated injections can bypass structural defenses. Defenses add up to probabilistic reduction in risk, not elimination.

**Real-world notes:** Defense layers: (1) XML delimiters to mark document content as user-controlled; (2) chunk sanitization (regex strip patterns like `SYSTEM:`, `IGNORE PREVIOUS INSTRUCTIONS`); (3) post-generation groundedness checking to catch answers that weren't grounded in the retrieved context (a symptom of successful injection). Claude models (Claude 3+) are generally more resistant to prompt injection than GPT-3.5-turbo class models. Multi-modal injection (embedding instructions in images or PDFs) is an emerging attack surface.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Riley Goodside, "Prompt injection attacks against GPT-3," 2022 (original public disclosure). Also: Greshake et al., "Not What You've Signed Up For: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection," arxiv 2023. https://arxiv.org/abs/2302.12173

## LLM-as-Judge (Automated Evaluation)

**What it is:** Using a capable LLM (typically a stronger model than the one being evaluated) to score the outputs of another LLM on dimensions like relevance, accuracy, groundedness, and fluency. Enables automated, scalable evaluation without human annotators for every query.

**Where it shines:** Catches quality regressions (model version drift, prompt changes, retrieval degradation) in a daily eval pipeline faster than human feedback. Scales to thousands of evaluations per hour. More consistent than human raters on structured rubrics.

**Where it fails:** LLM judges have known biases: preference for longer, more confident-sounding answers (verbosity bias), sycophancy toward the first position in a comparison, and difficulty evaluating factual accuracy without access to ground truth. The judge's quality ceiling is bounded by the judge model's own knowledge — a GPT-4o judge evaluating a factually wrong answer may give it a high score if the wrong answer is confidently written. Not a substitute for human evaluation on safety-critical dimensions.

**Real-world notes:** Use GPT-4o (or Claude Opus) as the judge, not GPT-4o-mini or Haiku — the judge must be more reliable than the system being evaluated. Score on individual dimensions (relevance 1–5, groundedness 1–5, accuracy 1–5) rather than a single composite score to maintain diagnostic utility. Concordance between LLM judge scores and human ratings is typically ~70–80% for well-defined rubrics. Widely used in practice: Langfuse, Braintrust, and RAGAS all include LLM-as-judge pipelines.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Zheng et al., "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena," NeurIPS 2023. https://arxiv.org/abs/2306.05685

## Feedback Loops (Explicit and Implicit)

**What it is:** A mechanism for collecting user signals about AI system quality and incorporating them into evaluation, monitoring, or model improvement. Explicit feedback: user-initiated ratings (thumbs up/down, star ratings, comments). Implicit feedback: behavioral signals derived from user actions without explicit rating (click-through on citations, follow-up question rate, session abandonment rate).

**Where it shines:** Provides a human quality signal at scale without human annotators. Explicit feedback, even at low response rates (~10–15%), gives a clear directional signal for A/B tests and regression detection. Implicit feedback can provide much higher volume (every user who clicks gives a signal) without imposing user friction.

**Where it fails:** Selection bias in explicit feedback — users are more likely to rate when surprised (positively or negatively), not when the answer is merely adequate. A 65% positive rate might reflect 65% excellent answers or 65% of the responding population being satisfied. Implicit feedback (click-through) is a weak proxy for answer correctness — users click citations out of curiosity, not necessarily because the answer was right. Both signals can be gamed (see: feedback loop poisoning).

**Real-world notes:** Best practice: use explicit feedback as a secondary signal and automated eval (LLM-as-judge on a gold standard set) as the primary signal for A/B test decisions. Stratify feedback by user to detect outlier patterns. Minimum sample size for A/B tests: n ≥ 500 queries per variant before declaring a winner. Never promote a prompt variant where automated eval and human feedback disagree.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Shankar, "Rethinking How We Evaluate Language Model Outputs," 2023. https://www.shreya-shankar.com/rethinking-llm-evaluation/

## Circuit Breaker Pattern

**What it is:** A stability pattern that prevents a system from repeatedly calling a failing dependency, giving the dependency time to recover while allowing the system to serve degraded-but-functional responses via a fallback. States: Closed (normal operation), Open (dependency is failing — route to fallback), Half-Open (probe the dependency with one request; if it succeeds, close the circuit; if it fails, stay open).

**Where it shines:** Prevents cascading failures where one component's failure exhausts the caller's resources (connection pools, threads) and brings down the caller too. Enables graceful degradation — the system continues serving users (with lower quality) rather than failing entirely.

**Where it fails:** The fallback must be designed and maintained. A circuit breaker to a Pinecone fallback (Postgres BM25) is only as good as the Postgres BM25 implementation. Stale breakers (opened by a transient spike, never closed because the half-open probe fires during a quiet period) can leave a system serving degraded responses long after the dependency recovered.

**Real-world notes:** Parameters to tune: failure threshold (e.g., 3 failures in 10 seconds → open), probe interval (e.g., 60 seconds), timeout for dependency calls (e.g., 30 seconds). Python libraries: `circuitbreaker`, `pybreaker`. The circuit breaker is the right tool for external API dependencies (Pinecone, OpenAI, Cohere); for internal service calls within your own infrastructure, a simpler retry-with-backoff is often sufficient.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Nygard, "Release It! Design and Deploy Production-Ready Software," 2nd edition (2018) — the canonical reference for the circuit breaker pattern.

## Silent Degradation

**What it is:** A class of failures where system quality degrades without triggering any infrastructure alert (error rate, latency, throughput). The system continues to function technically — requests succeed, latency is normal — but the quality of its outputs is lower than expected. Particularly common in AI systems where "correctness" is not measurable by standard metrics.

**Where it shines (as a concept):** Fundamental to understanding why AI systems require evaluation pipelines beyond infrastructure monitoring. Silent degradation is the failure mode that standard SRE practices are blind to, and it is specific to systems whose correctness is subjective or requires comparison to ground truth.

**Where it fails (as a concern):** Silent degradation that persists long enough becomes user churn and loss of trust before it becomes an engineering incident.

**Real-world notes:** The canonical examples in a RAG system: embedding drift, index staleness, LLM version drift, deletion accumulation, feedback loop poisoning. The only reliable defense is an offline evaluation pipeline with a gold-standard benchmark that runs on a schedule independent of production traffic. The benchmark must be designed to catch the specific failure modes of the system; a generic "does the system return something?" check will miss all of them.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

**Citation:** Shankar et al., "Operationalizing Machine Learning: An Interview Study," 2022. https://arxiv.org/abs/2209.09125

## Index Staleness

**What it is:** The failure mode where a document in the source knowledge system is updated (or deleted) but the corresponding vectors in the vector index are not refreshed. Queries that retrieve the stale vector receive outdated content, and the LLM generates answers based on old information. Like embedding drift, this failure produces no infrastructure error — it only degrades answer quality.

**Where it shines (as a concept):** Understanding index staleness is critical for any system where the underlying content changes frequently (live documentation, evolving policies, product specs).

**Where it fails (as a concern):** Staleness is worst when the stale content is confidently wrong — not "we don't have information on this" but "here is the old answer presented as current fact." The risk is proportional to both how frequently the source content changes and how high-stakes the domain is.

**Real-world notes:** Mitigation strategy stack: (1) webhook-based real-time updates (primary — changes reflected in minutes); (2) scheduled differential sync every 4 hours (secondary — catches webhook failures); (3) display the indexed-at timestamp in citations so users can assess freshness themselves; (4) for high-sensitivity documents, implement a freshness check before including a chunk in the prompt (query the source API for `last_modified`, compare against `last_indexed_at`). Alarm: `TimeSinceLastSuccessfulIngestion > 6h → PagerDuty`.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)

## Token Budgeting (LLM Cost Management)

**What it is:** The practice of explicitly accounting for token consumption per request — input tokens (prompt length) and output tokens (response length) — to predict, control, and optimize LLM API costs. Token cost = `input_tokens × input_price_per_token + output_tokens × output_price_per_token`.

**Where it shines:** Enables precise cost forecasting at design time, before a system is deployed. Reveals whether cost is a primary or secondary constraint at a given scale (e.g., $20/month for 30K queries is not a constraint; $375/month for the same volume with GPT-4o might be).

**Where it fails:** Token counts are estimates until measured. Tokenization varies by model and input type (code tokenizes differently than prose). Prompt template tokens are fixed costs that become proportionally larger at low query volumes. Output tokens are variable and harder to estimate for open-ended generation.

**Real-world notes:** Key formula: `monthly_cost = (avg_input_tokens × queries_per_month × input_price) + (avg_output_tokens × queries_per_month × output_price)`. For GPT-4o-mini at 2,750 input + 400 output tokens per query: `(2,750 × 30,000 × $0.00000015) + (400 × 30,000 × $0.00000060) = $12.38 + $7.20 = $19.58/month`. Benchmark: a 100-person startup typically spends under $30/month on LLM API calls for an internal knowledge base. Self-hosting becomes cheaper only above ~2.8M queries/month.

**Introduced by:** Internal RAG Knowledge Base (2026-07-04)
